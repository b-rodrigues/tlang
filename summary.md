# LLM Context: The T Programming Language

T is a reproducibility-first, functional programming language for data analysis. Its center of gravity is not “scripts that happen to run”, but **Nix-managed, pipeline-structured, polyglot analyses** that remain reproducible over time.

This file is an **LLM-oriented operating manual** for the repository. It is intentionally opinionated: if you generate T code, package code, documentation, or examples, follow the rules and conventions here.

## Non-Negotiable Principles

- **Nix is mandatory.** Treat Nix as the only supported installation and environment-management story for T.
  - End users run T with `nix shell github:b-rodrigues/tlang`.
  - Contributors develop T itself with `nix develop` and then `dune build` / `dune runtest`.
  - T projects and packages are Nix flakes; `flake.nix` and `flake.lock` pin exact environments.
  - Nix is also the build automation and sandboxing layer for pipelines, cross-language nodes, and Quarto rendering.
- **Pipelines are mandatory for reproducible, non-interactive work.** The normal shape of a serious T program is `pipeline { ... }`, not an ad hoc imperative script.
- **Polyglot execution is first-class.** T can orchestrate native T nodes, R nodes, Python nodes, shell/Bash nodes, and Quarto report nodes.
- **Errors are values, not exceptions.** `|>` short-circuits on `Error`; `?|>` forwards errors so recovery code can inspect them.
- **Missing values are explicit.** `NA` does not silently propagate through aggregations; functions typically require `na_rm = true` when missing values should be dropped.
- **Immutability is the default.** T code should be expression-oriented and pipeline-oriented, with no loops and no mutable variables.
- **Data-first APIs matter.** Standard-library APIs are designed so the data argument comes first, making them pipe-friendly.
- **Standard packages are auto-loaded.** You usually `import` user packages or local `.t` files; the built-in packages are already available in every session.
- **Intent blocks are part of the language design.** T is explicitly designed to support auditable human/LLM collaboration via `intent { ... }` metadata.

## Required Workflows

### Install and run T

```bash
nix shell github:b-rodrigues/tlang
```

That is the preferred way to get the `t` executable as a user. Do **not** suggest Homebrew, pip, conda, npm, curl installers, or hand-managed OCaml toolchains as alternatives.

### Start a new reproducible workspace

```bash
t init --project   # data-analysis project / pipeline workspace
t init --package   # reusable T package / library
```

Use **projects** for analyses and pipelines. Use **packages** for reusable exported functions.

### Enter the project or package environment

```bash
nix develop
```

Inside that shell, the project-specific T binary, dependency set, editor tooling (`t-lsp`), and pipeline runtimes are on `PATH`.

### Sync dependencies

- Projects declare dependencies in `tproject.toml`.
- Packages declare dependencies and metadata in `DESCRIPTION.toml` or project/package metadata generated by the scaffolder.
- After edits, run:

```bash
t update
```

That regenerates or syncs the Nix flake inputs so the environment and pipeline sandboxes stay reproducible.

### Common commands

```bash
t repl
# or simply: t

t run path/to/script.t

t test

t doc --parse --generate

t doctor
```

### Developing the compiler/runtime itself

```bash
nix develop
dune build
dune runtest
```

## Syntax and Semantics That Matter for LLMs

### Core syntax

```t
x = 10
name = "Alice"
add = \(a, b) a + b
result = if (x > 5) "high" else "low"

[1, 2, 3] |> map(\(v) v * v) |> sum

import mypackage
import mypackage[fn1, fn2]
import mypackage[alias = original]
import "helpers.t"[clean_data, normalize]

seq(5)
seq(1, 10, by = 2)

-- Pattern matching
match([1, 2]) { [h, ..t] => h, [] => 0 }
```

### Three different literal/block forms

- `[a, b, c]` = List
- `[key: value, ...]` = Dict
- `{ ... }` = Block syntax, **not** a Dict

### Pipe semantics

- `x |> f(...)` passes `x` as the **first** argument.
- `|>` stops immediately if the left-hand side is an `Error`.
- `?|>` forwards the left-hand side even if it is an `Error`, which is how recovery logic is written.

### Non-standard evaluation (NSE)

Inside `colcraft` verbs, use `$col` references:

```t
df
  |> filter($age >= 18)
  |> group_by($segment)
  |> summarize($avg = mean($value, na_rm = true))
```

For T code generation, assume `$col` syntax is the correct style for column references in data verbs.

### Datetime caveat

Treat `chrono::with_tz()` and `chrono::force_tz()` as **label-oriented in the current implementation**. Do not assume full timezone offset conversion unless the surrounding code or docs explicitly account for that limitation.

### Errors and NA

- Construct errors with `error("message")` or `error("Code", "message")`.
- Inspect them with `is_error(x)`, `error_code(x)`, `error_message(x)`, and `error_context(x)`.
- Missingness is explicit via `na()`, `na_int()`, `na_float()`, `na_bool()`, and `na_string()`.

### Imports and visibility

- `import pkg` imports all public names.
- `import pkg[name1, name2]` imports specific names.
- `import pkg[alias = original]` imports with aliases.
- Package functions are public by default; `@private` in T-Doc hides helper functions from importers.

### Intent blocks

```t
intent {
  description: "Analyze customer churn",
  goal: "Identify high-risk groups",
  assumptions: ["Churn = no purchase in 90 days"]
}
```

Use intent blocks to capture assumptions, constraints, required columns, outputs, and other LLM-relevant metadata.

## Pipelines, Runtimes, and Reproducibility

### Pipeline basics

```t
p = pipeline {
  raw = node(command = read_csv("data.csv"), runtime = T)
  clean = raw |> filter($age > 18)
}

build_pipeline(p)
```

Pipelines are DAGs. Nodes can be declared in any order; dependencies are resolved automatically.

### Node constructors and runtimes

- `node(...)` = fully configurable node constructor
- `rn(...)` = R node wrapper
- `pyn(...)` = Python node wrapper
- `shn(...)` = shell/Bash node wrapper
- `node(..., runtime = Quarto)` = Quarto render node

For Bash-specific syntax, prefer `shn(..., shell = "bash", shell_args = ["-lc"])`.

Useful patterns:

```t
p = pipeline {
  model_r = rn(script = "train.R", serializer = "pmml")
  model_py = pyn(script = "train.py", serializer = "pmml")
  report = shn(script = "postprocess.sh")
  quarto_report = node(script = "analysis.qmd", runtime = Quarto)
}
```

### Data interchange and sandboxes

- **Arrow** is the preferred interchange for DataFrames.
- **PMML** is used for model interchange.
- **JSON** is available for simpler values.
- Nodes run in Nix-managed sandboxes, so R/Python/shell/Quarto dependencies must be declared in the project/package metadata and synced via `t update`.

### Quarto specifics

- Add Quarto to `[additional-tools]` in the project/package config (for example `tproject.toml`) and run `t update`.
- T provisions the `tlang` Quarto extension into the project.
- In `.qmd` documents, use `read_node("name")` to consume pipeline artifacts.
- Use `pipeline_copy()` to bring rendered artifacts out of the Nix store and into the local workspace.

## Projects, Packages, Docs, and Editors

### Projects vs packages

- **Project** = analysis workspace, data, scripts, pipelines, pinned environment.
- **Package** = reusable library, exports functions, ships tests and docs.

Typical project files:
- `tproject.toml`
- `flake.nix`
- `flake.lock`
- `src/` or `scripts/`
- `data/`
- `_pipeline/`

Typical package files:
- metadata file(s)
- `flake.nix`
- `src/`
- `tests/`
- `docs/`

### T-Doc and generated reference docs

Source-level documentation uses `--#` comments. The generated reference lives in `docs/reference/` and `docs/reference/index.md`. If you need per-function detail beyond the compact index below, consult that generated reference.

### Editor support

- The language server is `t-lsp`.
- It is expected to be launched from inside `nix develop` (or via `direnv use flake`).
- Supported editor integrations are documented for VS Code/Positron, Vim/Neovim, Emacs, and Quarto workflows.

## Standard Packages and Signature Index

All standard packages are auto-loaded. Each package lives under `src/packages/<package>/`.

### `explain`

Purpose: value introspection and intent-block inspection.

- Introspection: `explain(x)`, `explain_json(x)`
- Intent helpers: `intent_fields(intent)`, `intent_get(intent, field)`

### `strcraft`

Purpose: string inspection, transformation, formatting, regex operations, and text layout helpers.

- Basic inspection and slicing: `str_nchar(s)`, `is_empty(s)`, `str_substring(s, start, end)`, `slice(s, start, end)`, `char_at(s, i)`, `index_of(s, sub)`, `last_index_of(s, sub)`, `contains(s, sub)`, `starts_with(s, prefix)`, `ends_with(s, suffix)`
- Replacement and normalization: `str_replace(s, pattern, repl)`, `replace_first(s, pattern, repl)`, `to_lower(s)`, `to_upper(s)`, `str_trim(s)`, `trim_start(s)`, `trim_end(s)`
- Formatting and joining: `str_repeat(s, n)`, `str_format(template, values)`, `str_sprintf(fmt, ...)`, `str_join(xs, sep = "")`, `str_string(x)`, `str_split(s, sep)`, `str_flatten(xs, collapse = "")`
- Regex/text-layout helpers: `str_lines(s)`, `str_words(s)`, `str_extract(s, pattern)`, `str_extract_all(s, pattern)`, `str_detect(s, pattern)`, `str_pad(s, width, side = "left", pad = " ")`, `str_trunc(s, width, side = "right", ellipsis = "...")`, `str_count(s, pattern)`

Note the overloaded names:
- `contains`, `starts_with`, and `ends_with` also exist as `colcraft` selection helpers.
- `slice` is both a `colcraft` row helper and a `strcraft` substring helper.
- `length` works on strings and collections in `core`, while `str_nchar` is the dedicated string-character-count helper.

Disambiguation rule of thumb for LLMs:
- inside `select(...)`/selection-helper contexts, prefer the `colcraft` helper meaning;
- with a `DataFrame` first argument, `slice(df, ...)` means row slicing;
- with `String` arguments, `contains(...)`, `starts_with(...)`, `ends_with(...)`, and `slice(...)` mean the `strcraft` string operations.

## Practical LLM Guidance

When generating T code, default to this shape:

1. Put reproducible work in `pipeline { ... }`.
2. Use `node`, `rn`, `pyn`, `shn`, and `runtime = Quarto` deliberately.
3. Keep the data argument first so `|>` reads naturally.
4. Use `$col` in `colcraft` verbs.
5. Handle `Error` and `NA` explicitly.
6. Assume Nix is responsible for installation, dependency resolution, build automation, and runtime isolation.
7. For projects, start with `t init --project`; for reusable libraries, start with `t init --package`.
8. For function-level details, consult `docs/reference/index.md` and the generated per-function reference pages.

## Release and Upgrade Workflows

### Releasing a new version of T (Contributors)

T uses a single source of truth for its version. To release a new version:

1.  **Bump the Version**: Update the version string in the root [VERSION](file:///home/brodrigues/Documents/repos/tlang/VERSION) file (e.g., `0.51.1`).
2.  **Sync Documentation**: Run the synchronization script to propagate the version to READMEs and documentation:
    ```bash
    ./scripts/sync_version.sh
    ```
3.  **Update Changelog**: Add the release notes and date to [docs/changelog.md](file:///home/brodrigues/Documents/repos/tlang/docs/changelog.md).
4.  **Commit and Push**:
    ```bash
    git add .
    git commit -m "chore: release 0.5x.x"
    git push origin main
    ```
    *The GitHub Action in `.github/workflows/release.yaml` will automatically create the Git tag and GitHub Release.*

### Upgrading a T project (Users)

To keep a T project updated with the latest language version and a fresh `nixpkgs` date:

```bash
t upgrade
```

This command automatically:
1.  Fetches the latest T version from GitHub.
2.  Retrieves today's date for the `rstats-on-nix` nixpkgs fork.
3.  Updates `tproject.toml` with the new values.
4.  Regenerates `flake.nix` and runs `nix flake update`.
