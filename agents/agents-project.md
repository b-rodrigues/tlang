# Agent Guide: T Project (Analysis & Pipelines)

This file provides critical context for AI agents (and human collaborators) working on this **T analysis project**. Following these rules ensures the project remains reproducible, auditable, and maintainable.

> [!IMPORTANT]
> **Mandatory Language Reference**: Before performing any tasks, you MUST read the **`T-LANGUAGE-REFERENCE.md`** file located in the project root. It contains essential syntax, standard library signatures, and pipeline patterns specific to this version of the T language.

---

## 1. The Environment (Nix-First)

**Nix is the only supported environment manager.** 
- **Activation**: You MUST work inside the shell provided by `nix develop`.
- **Dependencies**: All project dependencies (R packages, Python libraries, CLI tools) are declared in `tproject.toml`.
- **Synchronization**: After modifying `tproject.toml`, run `t update` to regenerate the Nix flake and update sandboxes.
- **Never** attempt to install packages via `install.packages()`, `pip install`, or `apt-get`.

---

## 2. Pipeline-First Architecture

T projects are not collections of ad-hoc scripts; they are **pipelines**.
- **The Pipeline Block**: Core logic belongs inside `pipeline { ... }`.
- **Node Granularity**: Prefer many small, single-purpose nodes over one large "do-it-all" script.
- **Polyglot Execution**: Use `rn()` for R, `pyn()` for Python, and `shn()` for Shell.
- **Artifact Access**: Use `read_node("node_name")` to consume data from upstream nodes and `pipeline_copy()` to bring final results into the local workspace.

---

## 3. Data Integrity and Interchange

- **Standard Interchange**: **Apache Arrow** is the default for DataFrames. Ensure nodes use `serializer = arrow` when passing tables between languages.
- **Model Interchange**: Use **PMML** for statistical models and **ONNX** for deep learning/ML models.
- **Raw Data**: Treat the `data/` directory as read-only. Never modify raw input files programmatically.
- **Schema Safety**: Use `$col syntax in colcraft verbs to ensure column references are correctly handled by T's non-standard evaluation.**

---

## 4. Coding Standards

- **Functional Style**: Use expression-oriented code. Avoid loops; use `map()`, `summarize()`, and `filter()`.
- **Immutability**: Variable reassignment is forbidden. Transform data through pipes (`|>`).
- **Error Handling**: T uses soft-fail semantics. Check for errors using `is_error()` and handle them via the error-forwarding pipe `?|>`.
- **Explicit Missingness**: Always handle `NA` values explicitly. Use `na_rm = true` in aggregations only when the logic requires dropping missing data.

---

## 5. Standard Project Structure

```text
.
├── tproject.toml       # Project manifest (Source of Truth)
├── flake.nix           # Auto-generated by tlang (Do not edit)
├── scripts/            # Pipeline definitions and node scripts (.t, .R, .py)
├── data/               # Raw data inputs (Read-only)
├── output/             # Final exported artifacts (via pipeline_copy)
└── tests/              # Project-level validation scripts
```

## 6. Common Commands for Agents

| Task | Command |
| :--- | :--- |
| **Sync Environment** | `t update` |
| **Run Analysis** | `t run scripts/pipeline.t` |
| **Run Tests** | `t test` |
| **Debug a Node** | `t explain --node name` |
| **Check Health** | `t doctor` |

---

## 7. Intent and Auditability

Always include `intent { ... }` blocks in your scripts to document:
- **Assumptions**: What must be true about the data?
- **Goals**: What is this specific script/node trying to achieve?
- **Dependencies**: Which columns or upstream nodes are required?
